# 工作說明 1 - 功能解析與調整報告

## [1] 採集物件 (`core/1_house.py`)
### x 和 w 欄位使用現狀
目前程式中涉及 **W 欄 (HF緯度)** 與 **X 欄 (HF經度)** 的地方如下：

1.  **`core/1_house.py`**:
    *   **資料建構**: 在 `row_data` 列表中，原本最後兩個元素 (索引 22, 23) 分別存放從網頁抓取的 `item['lat']` 與 `item['lng']`。
    *   **更新邏輯**: 在 `update_cells` 區塊中，原本會強制將這兩個值更新至試算表的第 23 欄 (W) 與第 24 欄 (X)。
2.  **`core/4_coords.py`**:
    *   **備援機制**: 程式會讀取「HF緯度」與「HF經度」作為「第三防線」。當仲介官網爬不到且 ArcGIS 也查不到座標時，會拿這兩個欄位當作備用座標。

### 已完成調整
*   已在 `core/1_house.py` 中將 W 和 X 欄位設為空字串，並註解掉強制更新試算表的邏輯，確保目前不再儲存這兩個欄位。

## [2] 去除重複 ID (`core/utils_deduplicate.py`)
### 去重邏輯與順序
程式採用 **「來源可靠性優先」** 的策略進行去重：

1.  **讀取順序**: 依據試算表由上而下的順序掃描。
2.  **ID 比對**: 以第一欄「物件ID」為基準。
3.  **仲介權重**: 當發現重複 ID 時，會比對 **V 欄 (來源仲介)**。
    *   優先序：**信義(1) > 永慶(2) > 好房(3) > 住商(4) > 591(5) ...**
    *   如果新掃描到的資料來源等級「高於」已存資料（數字較小），則會用新資料**取代**舊資料。
4.  **結果**: 最終每個 ID 只會保留一筆「最可靠來源」的資料。

## [3] 地址補完 (`core/2_address.py`)
### 補完方式
透過 Playwright 自動化瀏覽物件詳細頁 (`ExploreHouseNew`)：

1.  **鎖定表格**: 尋找包含「區域/門牌」與「坪數」關鍵字的資料表。
2.  **黃色列優先**: 系統會偵測背景顏色，優先抓取被網頁標示為**黃色**（目標案件）的那一列。
3.  **智能提取**:
    *   從單元格中過濾掉「已接委託」、「已售」等雜訊。
    *   結合「行政區」與「門牌路徑」組成完整地址。
4.  **輸出**: 寫入 **M 欄 (查地址)**。

## [4] 戶籍/座標擷取 (`core/3_registry.py`)
### 擷取方式
深入地籍資料頁面進行精準提取：

1.  **定位區塊**: 使用 `TreeWalker` 技術精確定位到 **「建築物所有權部」** 區塊。
2.  **欄位掃描**: 尋找「住址」或「住所」欄位。
3.  **OCR 支援**: 
    *   若住址是文字，直接抓取。
    *   若住址被圖片化（防爬機制），則自動啟動 **EasyOCR** 進行影像辨識。
4.  **地址清理**: 移除「里/鄰/樓層」等干擾資訊，並標準化縣市名稱（如：台北縣 ➔ 新北市）。
5.  **輸出**: 寫入 **O 欄 (戶籍地址)**。

---

## [5] 補齊座標 (純補) (`core/4_coords.py`)
### 補齊方式
採用「三道防線」確保座標精準度：

1.  **第一防線 (仲介官網)**: 若來源是信義或永慶，直接去該物件的仲介網頁抓取其內建的精準經緯度。
2.  **第二防線 (ArcGIS)**: 若第一防線失敗，將清理後的地址丟給 **ArcGIS Geocoding API** 查詢。
3.  **第三防線 (HF 備援)**: 若前兩者皆失敗，才使用原本抓取的 HF 座標（即上述 W/X 欄）。
4.  **輸出**: 分別寫入 **R, S 欄 (物件座標)** 與 **T, U 欄 (戶籍座標)**。
